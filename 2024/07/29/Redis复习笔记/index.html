1. 基础知识点

1.1 Redis与Memcached

共同点：

  1. 基于内存
  2. 有过期策略
  3. 性能高

不同点：

  1. Redis支持的数据类型丰富，但Memcached只支持最基础的key-value
  2. Redis支持数据持久化，可将内存数据保存到磁盘；但Memcached没有，重启或挂掉之后数据就丢失
  3. Redis原生支持集群，Memcached对集群没有原生支持
  4. Redis支持发布订阅，Lua脚本，事务等，Memcached不支持

1.2 LRU和LFU

LRU：最近最少使用，将最新操作的键移到链表头部，从链表尾部删除

LFU：最近最不常用，根据访问次数淘汰数据

2. 数据类型

2.1 String

2.1.1 实现

Redis中的String使用SDS进行实现

与C语言的String相比：

  1. SDS不仅能保存文本数据，也可以保存图片，视频等二进制数据
  2. SDS获取长度复杂度为O(1)，因为保存了长度变量
  3. SDS的API是安全的，拼接字符串不会导致缓冲区溢出

具体实现按照具体保存的值分三种情况：

  1. 整数值

    将encodeing设置为INT，且将整数值保存在ptr中

  2. 短字符串（在redis5.0及之后，短字符串是44子节及以下的数据）

    将encoding设置为embstr，且只需要调用一次内存分配函数分配redisObject和SDS的内存空间

  3. 长字符串

    将encoding设置为raw，且需要分配两次内存

2.1.2 常用命令

SET,GET,MEST,MGET,INCR,INCRBY,DECR,DECRBY

设置过期时间：EXPIRE(EX,秒级)，PEXPIRE(PX,毫秒级)

不存在则插入：SETNX

2.8 GEO

2.8.1 概要介绍

存储地理位置信息，并对存储的数据进行操作

2.8.2 实现

GEO 本身并没有设计新的底层数据结构，而是直接使用了 Sorted Set 集合类型。

GEO 类型使用 GeoHash 编码方法实现了经纬度到 Sorted Set 中元素权重分数的转换，这其中的两个关键机制就是「对二维地图做区间划分」和「对区间进行编码」。一组经纬度落在某个区间后，就用区间的编码值来表示，并把编码值作为 Sorted Set 元素的权重分数。

这样一来，我们就可以把经纬度保存到 Sorted Set 中，利用 Sorted Set 提供的“按权重进行有序范围查找”的特性，实现 LBS 服务中频繁使用的“搜索附近”的需求。

3. Redis线程模型

Redis采用单线程处理请求，但在启动时会额外启动三个后台线程用来处理耗时操作：关闭文件，AOF刷盘，释放内存

在Redis7.0及之后的版本中，开始支持多线程来处理网络I/O请求，但对于命令的处理仍然使用单线程

3.1 Redis使用单线程速度快的原因

  1. Redis大部分操作在内存中完成，因此CPU不是性能瓶颈
  2. 避免多线程竞争，避免线程切换的开销，且不会导致死锁
  3. 采用了多路复用机制

4. Redis持久化

4.1 AOF日志

每执行一条写操作命令，就把该命令以追加的方式写入到一个文件里

4.1.1 执行和写的顺序？

先执行后写

优势：

  1. 不用对命令进行检查，避免额外检查开销
  2. 当写操作执行成功之后才会写入AOF日志，因此不会阻塞当前写操作的执行

劣势：

  1. 导致数据丢失（没来得及保存到日志时宕机）
  2. 不阻塞当前操作，但是

  • 由内核决定何时数据写入硬盘

4.1.3 AOF重写

目的：解决AOF日志文件过大的问题

方式：读取当前数据库中的所有键值对，然后将每一个键值对用一条命令记录到「新的 AOF 文件」，等到全部记录完后，就将新的 AOF 文件替换掉现有的 AOF 文件

AOF重写是由子进程完成的，好处有两点：

  1. 不阻塞主进程
  2. 使用子进程而非子线程，因为线程是共享内存的，修改数据时需要加锁以保证数据安全。但使用子进程，父子进程以只读的方式共享内存，当修改的时候触发写时复制，进程获得独立的数据副本，在没有锁的情况下保障了数据安全

AOF重写过程中，主进程是可以修改数据的，导致了数据不一致，Redis解决方式如下：

  1. 在常见子进程同时创建一个AOF重写缓冲区
  2. 主进程在AOF重写期间执行的命令追加到重写缓冲区
  3. 子进程重写完毕后向主进程发送信号
  4. 将重写缓冲区的内容追加到新AOF日志文件中
  5. 覆盖现有AOF文件

4.2 RDB快照

优势：相对于AOF日志重新加载时需要把所有的指令全部执行一遍，RDB快照只需要把全部数据读入内存即可

4.3 混合持久化

在重写过程中，先是以RBD快照形式存到AOF日志文件中，然后对于主进程的增量命令采用AOF重写缓冲区解决，使用AOF日志形式进行持久化

5. 集群

5.1 主从复制

主服务器可以进行读写操作，当发生写操作时自动将写操作同步给从服务器，而从服务器一般是只读，并接受主服务器同步过来写操作命令，然后执行这条命令

5.1.1 主从复制第一次同步

第一阶段：

  • 从服务器向主服务器发送psync命令，带有两个参数：
    • runID：每个redis服务器启动时自动生成的ID，这里本应该表示主服务器ID，但是从服务器并不知道主服务器ID，所以为”?”
    • offset：复制的进度，第一次同步时进度为-1
  • 主服务器返回FULLRESYNC命令，带着自己的runID和offset

第二阶段：

  • 主服务器使用bgsave生成RDB文件，并用子进程发送给从服务器

第三阶段：

  • 在主服务器生成、发送RDB文件，以及从服务器加载RDB文件时，主服务器可能执行新命令，这些新命令将被保存到replication buffer缓冲区中
  • 从服务器加载完毕后会向主服务器发送确认消息
  • 主服务器收到确认消息后，将replication buffer中的命令发送给从服务器，完成第一次数据同步

5.1.2 命令传播

第一次同步之后，主从服务器之间维持一个TCP长连接，后续数据通过该连接进行发送

5.1.3 分摊主服务器压力

从服务器也可承担主服务器的责任

5.1.4 增量复制

整个集群维护一个环形缓冲区repl_backlog_buffer

replication offset用于在repl_backlog_buffer上标注缓冲区的同步进度，每个节点都有一个进度，主服务器的叫master_repl_offset，记录自己写的位置；从服务器的叫slave_repl_offset，记录自己读的位置

增量复制通常用于连接断开后重连的情况，根据主从读写偏移量计算出主服务器需要发送哪些数据给从服务器

缓冲区数据覆盖：

对同一个节点来说，这个进度是一直递增的，但是缓冲区是环形且大小有限的，因此比如说在缓冲区有地址A和B，B在A之后。现在主服务器写到B了，A的偏移是10，B的偏移是15。此时从服务器的读取偏移量是9，马上就要读A了，但是这时候从服务器的连接断了，等它再连回来的时候，主服务器已经转过一圈并且把A地址的偏移量改成30了，这时候A想读的数据已经被覆盖了

对于数据覆盖的情况，需要直接使用全量复制

因此，需要合理适当增大环形缓冲区容量

5.2 哨兵模式

5.2.1 判断节点故障

主观下线：一个哨兵在给某个节点PING之后没在规定时间内收到回复，那么该哨兵就认为该节点主观下线

客观下线：一个哨兵判断某节点主观下线之后，会发起赞同/反对投票。当获得的赞同票数达到配置的quorum数量时，则该哨兵认为该节点客观下线。一般quorum配置为：哨兵数/2+1

5.2.2 选举哨兵Leader

候选者：判断节点客观下线的哨兵都是候选者

选举：

  1. 每个哨兵只有一次投票机会，且候选者将票投给自己
  2. 候选者要求其它哨兵给自己投票
  3. 候选者完成以下两个条件即可成为leader：1.得到半数以上赞成票 2.票数大于等于quorum

5.2.3 哨兵数量

最少为3个，否则只要一个哨兵宕机，就无法得到半数以上票

5.2.4 主从故障转移

  1. 在旧主节点下属所有从节点中选择一个节点设为新主节点，选择依据为：优先级>复制进度靠前>ID号小
  2. 将从节点指向新主节点
  3. 通知客户端主节点更换（发布/订阅机制）
  4. 监控旧主节点，待恢复后将其变为从节点

5.3 切片集群模式

主要是当数据量过大，一个节点存储不下的时候，使用集群来分担数据存储

原理是根据存储数据的key，哈希到对应哈希槽中；每个哈希槽都被分配(平均分配/手动分配)在某一个节点上

6. Redis缓存设计

6.1 缓存雪崩

描述：由于缓存中大批量数据同一时间过期，查询数据量巨大，均需要访问数据库，导致数据库负载过大最终崩溃；强调大量不同数据

解决方案：

  1. 将缓存失效时间打散
  2. 设置缓存不过期

6.2 缓存击穿

描述：由于某热点数据过期，庞大用户量访问该热点数据导致数据库负载过大最终崩溃；强调对同一个数据的大量并发

解决方案：

  1. 互斥锁，保证同一时间只有一个业务线程请求缓存
  2. 不给热点数据设置过期时间，后台异步更新缓存；或者热点数据过期前提前通知后台线程更新缓存且重设过期时间

6.3 缓存穿透

描述：大量并发查询缓存和数据库中均不存在的数据

解决方案：

  1. 限制非法请求
  2. 发生缓存穿透现象时，在缓存中针对该key设置默认值或空值
  3. 布隆过滤器

6.4 缓存更新策略

6.4.1 旁路缓存策略

应用程序直接与数据库、缓存交互，并负责缓存的维护

写策略：先写数据库，后删除缓存

读策略：若命中缓存则直接返回；若未命中缓存则从数据库中读后更新缓存

*需要注意写策略必须先写后删：理论上说，无论先写后删还是先删后写，都会在并发的情况下有数据库缓存的不一致性风险；但是由于对缓存的操作要远快于对数据库的操作，先写后删更能降低不一致性风险（有两个进程A和B，A先修改数据库设置name=”ming”，这时候同时发生**A修改缓存name=”ming”和B修改数据库name=”zhang”*这两件事，因为修改缓存远快于修改数据库，所以最后一定是B修改缓存name=”zhang”，确保了一致性。如果过程反过来就无法保证）

适用：读多写少的场景，不适用于写多的场景

6.4.2 读穿/写穿策略

客户端只与缓存交互，数据库相关的操作全部交给缓存

该策略比较少见

6.4.3 写回策略

写操作只操作缓存，将缓存数据设置为脏并立马返回，不会立马更新数据库

对于数据库，采用批量异步更新

适用：写多

缺点：容易造成数据丢失，数据不是强一致性的
