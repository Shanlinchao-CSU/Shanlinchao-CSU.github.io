一、Java集合

1.1 Collection接口

  • List
    • ArrayList：底层结构为数组，可以动态扩容
    • LinkedList：底层结构为链表，内存空间不连续，因此不支持快速随机访问
      • ArrayList&&LinkedList：ArrayList由数组实现，随机访问快，但插入删除效率较低；LinkedList相反
    • Vector：List的古老实现，线程安全
  • Set
    • HashSet：底层就是HashMap，key存数值，value存固定值。元素无序且唯一
    • LinkedHashSet：底层就是LinkedHashMap，元素有序且唯一
    • TreeSet：底层采用红黑树实现，元素有序且唯一
  • Queue
    • PriorityQueue：优先队列，堆实现
    • DelayQueue：延迟队列，队列中元素需实现Delayed接口，且按过期时间排序。越快过期的元素越靠队头；支持阻塞获取元素
    • ArrayDeque：双端队列，两端都允许插入弹出

1.2 Map接口

  • Map
    • HashMap：数组+链表/红黑树
    • LinkedHashMap：HashMap+双向链表，在HashMap的基础上维护了键值对的顺序
    • HashTable：数组+链表
      • HashMap&&HashTable：
        1. HashMap线程不安全，HashTable线程安全
        2. HashMap效率高，HashTable效率低
        3. HashMap可接受NULL键值，HashTable不可接受
        4. HashMap继承AbstractMap，HashTable继承Dictionary
        5. HashMap底层数组初始容量16，每次扩容容量为2n，若指定容量，则初始容量会是2的幂；HashTable的是11，每次扩容容量为2n+1，若指定容量，初始容量就是指定的
        6. HashMap对哈希值高位低位进行了混合扰动以减少冲突，而HashTable直接使用键的hashCode()值
    • TreeMap：红黑树

1.3 零碎知识点

1.3.1 ArrayList扩容机制

扩容可分为两种情况：

  • 当ArrayList的容量为0时，此时添加元素的话，需要扩容，三种构造方法创建的ArrayList在扩容时略有不同：

    1. 无参构造，创建ArrayList后容量为0，添加第一个元素后，容量变为10，此后若需要扩容，则正常扩容
    2. 传容量构造，当参数为0时，创建ArrayList后容量为0，添加第一个元素后，容量为1，此时ArrayList是满的，下次添加元素时需正常扩容。
    3. 传列表构造，当列表为空时，创建ArrayList后容量为0，添加第一个元素后，容量为1，此时ArrayList是满的，下次添加元素时需正常扩容

  • 当ArrayList的容量大于0，并且ArrayList是满的时，此时添加元素的话，进行正常扩容，每次扩容到原来的1.5倍

1.3.2 Comparable和Comparator

  1. Comparable表示“能够比较”，一般自定义的，需要实现比较的类会实现该接口，在自定义类内部实现比较具体逻辑；Comparator是一个比较器，用于在外部排序方法对某个类进行比较
  2. 实现Comparable需要实现compareTo方法；实现Comparator需要实现compare方法

1.3.3 Queue和Deque

Queue是单端队列，而Deque是双端队列，两头都可以插入弹出

1.3.4 ArrayDeque和LinkedList

这俩都实现了Deque接口，但：

  1. 底层实现不同：前者使用数组+双指针实现，后者使用链表实现
  2. 前者不支持NULL，但后者支持
  3. 前者效率比后者高

1.3.5 HashMap和TreeMap

TreeMap额外继承了NavigableMap和SortedMap接口

前者保证了TreeMap对元素的搜索能力

后者保证了TreeMap对元素的排序能力

1.3.6 为何HashMap数组长度一定要2的n次方？

因为hash%length == hash&(length-1)的前提是 length 是 2 的 n 次方，length为2的n次方之后，取余操作就可用位运算代替，速度更快

原理：假设length=2^n，那么hash/length等价于hash除以n个2，也就是将hash向右移动n位，所以取余的结果就是被移掉的n位，也就是hash的低n位，hash&(length-1)就是计算这n位的

1.3.7 HashMap的线程不安全

  • 死循环

    JDK1.7及以前，由于链表采用头插法，多线程环境下可能导致指针错误，导致死循环；但JDK1.8及以后改为尾插法，问题不存在

  • 数据覆盖

    这个是一直有的问题，一个线程可能把另一个线程的数据覆盖

3.2 垃圾回收

堆是垃圾收集器管理的主要区域

大对象直接进入老年代

长期存活对象进入老年代

  • 一个对象每熬过一次垃圾回收，则岁数+1，当岁数大于某个阈值时，则会被放到老年代中

3.2.1 死亡对象判断方法

  • 引用计数法

    以被引用次数作为判断对象死亡的依据，但很难解决循环引用的问题，因此基本不用

  • 可达性分析法

    从一个根节点向下搜索，不在引用链上的对象就是垃圾对象

    根节点的选择：虚拟机栈、本地方法栈中引用的对象；元空间中类静态属性、常量引用的对象等

3.2.2 垃圾收集算法

  • 标记清除算法

    存在问题：效率低、产生碎片空间

  • 复制算法

    每次只使用一半的空间，使用完了就把存活的对象复制到另外半片空间

    因为要复制所有存活的，所以适用于新生代

  • 标记整理算法

    标记后，将存活的对象移到一起，同时清除剩下的空间

  • 分代收集算法

    对于不同代，采用不同的垃圾收集算法

    对于新生代，存活周期短，适合使用复制算法

    对于老年代，存活周期长，适合使用标记整理算法

3.2.3 垃圾收集器

  • Serial收集器

    单线程的收集器，进行垃圾回收时需要暂停所有的其它工作线程，直到其工作结束

  • ParNew收集器

    垃圾回收过程使用了多线程实现，其余与Serial相同

  • Parallel Scavenge收集器

    与ParNew收集器非常相似，Parallel Scavenge收集器追求最大化CPU吞吐量

  • Serial Old收集器

    Serial收集器的老年版本

  • Parallel Old收集器

    Parallel Scavenge收集器的老年版本

  • CMS收集器

    追求最小化用户线程停顿时间，实现了垃圾收集与用户线程的同时进行，是并发收集器

    CMS只回收老年代和永久代，不回收年轻代

    原理

    • 初始标记：停止用户线程，寻找老年代中所有的GC Root对象和被年轻代中存活的对象所引用的老年代的对象
    • 并发标记：开启用户线程，从第一步所标记的对象开始标记可达对象；期间如果有对象的引用发生改变，就将该对象的Card标记为Dirty
    • 重新标记：停止用户线程，扫描Dirty Card的所有对象并进行可达性分析，对第二步遗漏的进行标记
    • 并发清除：开启用户线程，对未标记区域进行清除

    缺点

    1. 对CPU资源敏感
    2. 无法处理浮动垃圾
    3. 使用标记-清除算法，产生大量碎片空间

  • G1收集器

    将堆内存直接划分为多个大小相等的区域，每块区域可根据需要充当Eden，Survivor或老年代区域

    允许使用者设置最大容忍用户线程停顿时间，并自动计算在时间内能收集几个区域

    原理

    • 初始标记
    • 并发标记
    • 最终标记：跟CMS的重新标记类似，只是范围更小
    • 筛选回收：跟CMS的并发清除类似，只是范围更小

    相比于CMS，G1由于将堆划分为多个大小相等的区域，以区域为单位进行处理，因此在更大内存中有更好的表现

    G1采用标记-整理算法，不会产生很多碎片，且由于是对区域操作，效率影响不大
